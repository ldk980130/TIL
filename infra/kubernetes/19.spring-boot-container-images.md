# Spring Boot — Container Images

- [공식 문서 - Container Images](https://docs.spring.io/spring-boot/reference/packaging/container-images/index.html)
- [공식 문서 - Efficient Container Images (Layering)](https://docs.spring.io/spring-boot/reference/packaging/container-images/efficient-images.html)
- [공식 문서 - Efficient Deployments (Unpacking the JAR)](https://docs.spring.io/spring-boot/reference/packaging/efficient.html)
- Spring Boot 앱을 컨테이너 이미지로 만드는 두 갈래 — **Dockerfile + layered jar** 직접 작성 / **Cloud Native Buildpacks**로 Dockerfile 없이 생성
- 두 방식의 공통 핵심 = **변경 빈도가 다른 파일을 다른 레이어로 분리** → Docker 레이어 캐시를 최대한 재사용 (앱 코드만 바뀌어도 의존성 레이어는 그대로)

## 실행 가능 JAR의 구조와 jarmode tools

- Spring Boot의 uber jar(=fat/nested jar)는 `BOOT-INF/lib/` 안에 의존성 jar를 **중첩**으로 품음 → 중첩 jar에서 클래스 로딩하는 **약간의 startup 비용** 발생
- `-Djarmode=tools` 시스템 프로퍼티로 jar를 **도구 모드**로 실행하면 내용물을 풀거나 레이어 정보를 조회

```bash
java -Djarmode=tools -jar app.jar list-layers   # 추출 가능한 레이어 목록
java -Djarmode=tools -jar app.jar extract        # 기본 layout으로 압축 해제
java -Djarmode=tools -jar app.jar extract --layers --destination extracted  # 레이어별로 분리 추출
```

- **`tools` jarmode는 구버전 `layertools`를 대체** (Spring Boot 3.3+) — 옛 튜토리얼의 `-Djarmode=layertools`를 봤다면 지금은 `tools`
- 기본 layout 추출 시 — 라이브러리는 `lib/`로, app jar에는 클래스 + `lib/`를 참조하는 manifest만 남음
- **프로덕션은 압축 해제(exploded)해서 실행 권장** — 중첩 jar 로딩 비용이 사라져 **기동이 빠름**(시작 이후 실행 성능은 동일). 기본 layout은 **AOT cache·CDS 친화적**

```bash
java -jar app/app.jar   # 추출된 디렉터리 구조에서 실행
```

## 레이어드 JAR — 변경 빈도로 레이어 분리

- `extract --layers`가 만드는 **4개 기본 레이어** (먼저 COPY되는 것 = 변경 빈도 낮은 것부터)

| 순서 | 레이어 | 내용 | 변경 빈도 |
| --- | --- | --- | --- |
| 1 | `dependencies` | 정식 릴리스 의존성 | 가장 낮음 → 최하단(캐시 재사용 극대화) |
| 2 | `spring-boot-loader` | `org/springframework/boot/loader` 로더 클래스 | 거의 안 바뀜 |
| 3 | `snapshot-dependencies` | SNAPSHOT 의존성 | 릴리스보다 자주 바뀜(분리해 릴리스 레이어 보호) |
| 4 | `application` | 앱 클래스·리소스(`BOOT-INF/classes`, manifest) | 가장 높음 → 최상단 |

- jar 안의 **`layers.idx`** 파일이 "어떤 파일이 어느 레이어에 속하는지" 매핑을 정의 → 추출·이미지 빌드가 이 인덱스를 따름
- **Spring Boot 2.4+부터 레이어링 기본 활성**. 끄거나 레이어 구성을 커스터마이징할 때만 빌드 플러그인 설정 사용

```xml
<!-- Maven: spring-boot-maven-plugin -->
<configuration><layers><enabled>true</enabled></layers></configuration>
```

```gradle
// Gradle
bootJar { layered() }
```

## Dockerfile로 빌드 — multi-stage + 레이어별 COPY

- **builder 스테이지**에서 `extract --layers`로 풀고, **runtime 스테이지**에서 레이어 디렉터리를 **각각 COPY** → COPY 한 줄당 Docker 레이어 1개

```dockerfile
FROM bellsoft/liberica-openjre-debian:25-cds AS builder
WORKDIR /builder
COPY target/*.jar application.jar
RUN java -Djarmode=tools -jar application.jar extract --layers --destination extracted

FROM bellsoft/liberica-openjre-debian:25-cds
WORKDIR /application
# 변경 빈도 낮은 레이어부터 COPY → 의존성 레이어는 캐시 재사용, 앱 코드만 바뀌면 마지막 레이어만 무효화
COPY --from=builder /builder/extracted/dependencies/ ./
COPY --from=builder /builder/extracted/spring-boot-loader/ ./
COPY --from=builder /builder/extracted/snapshot-dependencies/ ./
COPY --from=builder /builder/extracted/application/ ./
ENTRYPOINT ["java", "-jar", "application.jar"]
```

- 캐시 효과 — `dependencies`는 거의 안 바뀌므로 push/pull 시 **변경된 레이어만 전송**, 이미지 빌드도 빨라짐
- 이미지 슬림화·레이어 재사용은 노드의 **imagefs 사용량·image pull 시간**을 줄여 [[node-pressure-eviction]](17번)의 디스크 압박 완화에도 유리
- 추가 최적화 — builder에서 **training run**을 한 번 돌려 기동 가속: Java 24+는 AOT cache(`-XX:AOTCacheOutput`/`-XX:AOTCache`), 이전은 CDS(`-XX:ArchiveClassesAtExit`/`-XX:SharedArchiveFile`)

## Cloud Native Buildpacks — Dockerfile 없이 이미지 생성

- Dockerfile을 직접 작성·관리하지 않고 빌드 명령 하나로 **OCI 이미지** 생성 후 로컬 Docker daemon에 push

```bash
./gradlew bootBuildImage      # Gradle
mvn spring-boot:build-image   # Maven
```

- **buildpack이 자동 처리** — jar를 감지해 적절한 JRE를 얹고 레이어를 구성(Cloud Foundry/Heroku buildpack과 같은 메커니즘). 결과물은 어떤 OCI 런타임에서도 실행 가능
- **Paketo Spring Boot buildpack이 `layers.idx`를 반영** → 위 레이어드 JAR 커스터마이징이 빌드팩 이미지에도 그대로 적용
- buildpack은 **재현 가능한 빌드(reproducible build)**·캐시를 위해 리소스의 "last modified" 타임스탬프를 조작 → 런타임에 그 메타데이터에 의존하면 안 됨

```properties
spring.web.resources.cache.use-last-modified=false   # 정적 리소스가 last-modified에 의존하지 않도록
```

## 언제 무엇을 쓰나

| 기준 | Dockerfile + layered jar | Cloud Native Buildpacks |
| --- | --- | --- |
| Dockerfile 관리 | 직접 작성·유지 | 불필요(빌드 명령만) |
| 베이스 이미지·JRE 제어 | 완전 제어(원하는 base 지정) | buildpack이 자동 선택 |
| 빌드 진입점 | `docker build` | `bootBuildImage` / `build-image` |
| 적합 | 세밀한 커스터마이징·기존 Docker 파이프라인 | 표준화·빠른 시작·Dockerfile 보일러플레이트 제거 |
