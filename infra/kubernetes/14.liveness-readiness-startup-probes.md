# Liveness, Readiness, Startup Probes

- [공식 문서 - Configure Liveness, Readiness and Startup Probes](https://kubernetes.io/docs/tasks/configure-pod-container/configure-liveness-readiness-startup-probes/)
- kubelet이 컨테이너 상태를 주기적으로 검사하는 메커니즘
- 세 종류 — **liveness**(죽었는지), **readiness**(트래픽 받을 준비 됐는지), **startup**(아직 초기화 중인지)

## Liveness Probe

- 오래 실행되는 앱이 **복구 불가능한 고장 상태**에 빠졌을 때 감지 → kubelet이 컨테이너를 **kill하고 재시작**
- 4가지 검사 방식 — exec / HTTP / TCP / gRPC

### exec

- 컨테이너 안에서 **명령을 실행** — 종료 코드 0이면 성공, 그 외는 실패

```yaml
livenessProbe:
  exec:
    command: ["cat", "/tmp/healthy"]
  initialDelaySeconds: 5   # 첫 probe까지 대기
  periodSeconds: 5          # 이후 검사 주기
```

- 동작 예시 — 컨테이너가 시작 후 30초 뒤에 `/tmp/healthy`를 삭제하면, `cat` 명령이 실패 → kubelet이 컨테이너를 kill하고 재시작
- `failureThreshold`(기본 3) 연속 실패해야 컨테이너가 kill됨

### HTTP

- kubelet이 컨테이너의 지정 엔드포인트에 **HTTP GET 요청** — 응답 코드 **200–399면 성공**, 그 외는 실패

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 3
  periodSeconds: 3
```

- 로컬 HTTP 프록시 환경 변수(`HTTP_PROXY` 등)는 liveness probe에 **영향을 주지 않음** (v1.13+)

### TCP

- kubelet이 컨테이너의 지정 포트에 **TCP 소켓 연결** 시도 — 연결 성공이면 healthy, 실패면 unhealthy

```yaml
livenessProbe:
  tcpSocket:
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
```

- HTTP와 설정 구조가 거의 동일 — 같은 Pod에서 **readiness와 liveness를 동시에** TCP로 설정할 수도 있음

### gRPC

- 앱이 [gRPC Health Checking Protocol](https://github.com/grpc/grpc/blob/master/doc/health-checking.md)을 구현한 경우 사용 (v1.27 stable)

```yaml
livenessProbe:
  grpc:
    port: 2379
  initialDelaySeconds: 10
```

- 주의사항
    - Pod IP 또는 hostname으로 실행됨 — gRPC 엔드포인트가 **Pod IP에서 listen**하도록 설정해야 함
    - **인증 파라미터 미지원** (`-tls` 등)
    - 모든 에러는 probe 실패로 처리됨 — 별도 에러 코드 구분 없음
    - **named port 미지원** (HTTP·TCP는 지원)

## Named port

- HTTP·TCP probe에서 `port`에 숫자 대신 **이름으로** 참조 가능

```yaml
ports:
- name: liveness-port
  containerPort: 8080

livenessProbe:
  httpGet:
    path: /healthz
    port: liveness-port    # 포트 번호 대신 이름
```
