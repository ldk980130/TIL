# 8. 카프카 트랜잭션과 exactly-once

## 트랜잭션 활성화 — transactionIdPrefix

- 프로듀서 팩토리에 `transactionIdPrefix`를 설정하면 트랜잭션이 켜지고 `KafkaTemplate` 작업이 트랜잭셔널해짐
    ```java
    pf.setTransactionIdPrefix("my.txid.");          // 또는 spring.kafka.producer.transaction-id-prefix
    ```
- Boot는 이 설정이 있으면 `KafkaTransactionManager`를 자동 구성하고 리스너 컨테이너에 연결
- 동작: 팩토리가 트랜잭셔널 프로듀서를 캐시로 관리, 각 프로듀서의 `transactional.id` = `prefix + n`
- ⚠️ 멀티 인스턴스 배포 시 `transactionIdPrefix`는 **인스턴스마다 유니크**해야 함
- 멱등성·정확히 한 번 전송 이론은 [5.3 중복 없는 전송](../practical-kafka/05.kafka-producer/5.3.deduplicated-transfer.md)·[5.4 정확히 한 번 전송](../practical-kafka/05.kafka-producer/5.4.exactly-once.md) 참고

## 로컬 트랜잭션 — executeInTransaction

- 트랜잭션 매니저 없이 여러 send를 한 트랜잭션으로 묶을 때
    ```java
    template.executeInTransaction(t -> {
        t.send("topic", "a");
        t.send("topic", "b");
        return true;       // 정상 종료 → 커밋, 예외 → 롤백
    });
    ```

## @Transactional — 프로듀서 트랜잭션

- `@Transactional` + `KafkaTransactionManager` → 메서드 안의 모든 send가 함께 커밋/롤백
    ```java
    @Transactional
    public void process(List<Thing> things) {
        things.forEach(t -> kafkaTemplate.send("topic", t));  // 메서드 정상 종료 시 일괄 커밋
    }
    ```
- 템플릿과 트랜잭션 매니저가 **같은 `ProducerFactory`**를 써야 함

## 컨테이너 트랜잭션 — read-process-write (EOS)

- 컨테이너에 `KafkaTransactionManager`가 연결되면, 소비한 레코드/배치마다 트랜잭션을 시작
- exactly-once 달성 방식
    - 리스너에서 보낸 레코드가 트랜잭션에 포함됨
    - 소비한 오프셋을 프로듀서의 `sendOffsetsToTransaction()`으로 같은 트랜잭션에 전송
    - → **오프셋 커밋 + 발행이 원자적** (둘 다 커밋되거나 둘 다 롤백) = consume-process-produce의 exactly-once
- `EOSMode.V2`(구 BETA)가 3.0+ 유일 지원 모드
- 소비 측 이론은 [6.5 정확히 한 번 컨슈머 동작](../practical-kafka/06.kafka-consumer/6.5.exactly-once-read-consumer.md) 참고

## 컨슈머 isolation.level

- 트랜잭션으로 쓰인 메시지를 **커밋된 것만** 읽으려면 컨슈머에 `isolation.level=read_committed`
    ```yaml
    spring:
      kafka:
        consumer:
          isolation-level: read_committed   # 미커밋/중단된 트랜잭션 메시지는 건너뜀
    ```

## DB 트랜잭션과 결합 — 트랜잭션 동기화

- `@Transactional`(DB 트랜잭션 매니저) 안에서 `KafkaTemplate.send`를 호출하면 DB 트랜잭션에 **동기화**되어 메서드 종료 시 함께 커밋
- 커밋 순서: 기본 **DB 먼저 → Kafka 나중**. Kafka-first가 필요하면 중첩 `@Transactional`로 안쪽을 `KafkaTransactionManager`로
- ⚠️ 두 트랜잭션은 진짜 원자적이 아님 — DB 커밋 후 Kafka 커밋이 실패하면 불일치 가능(**이중 쓰기 문제**). 완전한 보장은 아웃박스(Outbox) 패턴 등 별도 설계 필요

## 동시성과 transactional.id 풀

- `ConcurrentMessageListenerContainer`에서 트랜잭션을 쓰면 동시 컨슈머마다 `transactional.id`가 필요 → `TransactionIdSuffixStrategy`의 `maxCache ≥ concurrency`로 풀 확보 (부족하면 `NoProducerAvailableException`)
