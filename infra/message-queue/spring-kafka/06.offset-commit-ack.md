# 6. 오프셋 커밋과 AckMode

## Spring의 오프셋 커밋 방식

- 네이티브 카프카는 `enable.auto.commit=true`로 주기적 자동 커밋이 기본 — 하지만 처리 완료 전에 커밋될 수 있어 유실 위험
- spring-kafka는 컨테이너가 **직접 커밋을 관리** — 2.3부터 `enable.auto.commit`을 명시 설정하지 않으면 무조건 `false`로 두고, `AckMode`에 따라 처리 완료 시점에 커밋
- 오프셋 개념·이론은 [6.1 컨슈머 오프셋 관리](../practical-kafka/06.kafka-consumer/6.1.consumer-offset.md) 참고

## AckMode 종류

| AckMode | 커밋 시점 |
|---|---|
| `RECORD` | 레코드 하나 처리(리스너 리턴) 후 |
| `BATCH` (기본) | `poll()`로 가져온 레코드 전부 처리 후 |
| `TIME` | 전부 처리 후 + 마지막 커밋 뒤 `ackTime` 경과 시 |
| `COUNT` | 전부 처리 후 + 마지막 커밋 뒤 `ackCount`개 도달 시 |
| `COUNT_TIME` | `TIME` 또는 `COUNT` 중 하나라도 충족 시 |
| `MANUAL` | 리스너가 `acknowledge()` 호출 후 — 단 BATCH 시맨틱으로 커밋 |
| `MANUAL_IMMEDIATE` | `acknowledge()` 호출 즉시 커밋 |

- 앞 5개는 컨테이너가 자동으로 커밋, 뒤 2개(`MANUAL`/`MANUAL_IMMEDIATE`)는 리스너가 `Acknowledgment`로 직접 커밋

## 수동 커밋 — Acknowledgment

- `MANUAL`/`MANUAL_IMMEDIATE`에선 리스너 파라미터로 `Acknowledgment`를 주입받아 직접 커밋
    ```java
    @KafkaListener(topics = "t", ackMode = "MANUAL_IMMEDIATE")
    public void listen(String data, Acknowledgment ack) {
        // ... 처리 ...
        ack.acknowledge();          // 커밋
    }
    ```
- ⚠️ 이름과 달리 `MANUAL`은 **즉시 커밋이 아님** — `acknowledge()`를 호출해도 poll 전체가 끝난 뒤 BATCH처럼 커밋됨. 호출 즉시 커밋하려면 `MANUAL_IMMEDIATE`
- 재전송용 음수 ack (블로킹 재시도의 대안, → 5단계)
    - `nack(Duration sleep)` — 레코드 리스너: 해당 레코드부터 다시 받음 (sleep 후 재전송)
    - `nack(int index, Duration sleep)` — 배치 리스너: index 레코드부터 다시
- `acknowledge(int index)` (배치, 3.0.10+, `MANUAL_IMMEDIATE` 필요) — index까지의 레코드만 커밋

## AckMode 설정

```java
factory.getContainerProperties().setAckMode(AckMode.MANUAL_IMMEDIATE);  // 팩토리/컨테이너 단위
// 또는 리스너 단위(4.1+): @KafkaListener(ackMode = "MANUAL_IMMEDIATE")
```

## 순서 없는 커밋 — asyncAcks

- 기본적으로 `MANUAL`/`MANUAL_IMMEDIATE` ack는 **순서대로** 커밋해야 함 (카프카는 파티션당 커밋 오프셋을 하나만 유지)
- `asyncAcks=true` (2.8+)면 poll 내 레코드를 **임의 순서로 ack** 가능 — 비동기 처리에 유용
    - 컨테이너가 누락된 ack를 기다렸다가 순서를 맞춰 커밋, 이전 poll이 다 커밋될 때까지 컨슈머 일시정지
    - 주의: 실패 시 **중복 전달 가능성↑**, `nack()` 사용 불가
