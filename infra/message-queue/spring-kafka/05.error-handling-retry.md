# 5. 에러 핸들링과 재시도·DLT

## 컨테이너 에러 핸들러 — DefaultErrorHandler

- 리스너 메서드에서 예외가 나면 컨테이너의 `CommonErrorHandler`가 처리
- 기본 구현은 `DefaultErrorHandler` (구 `SeekToCurrentErrorHandler`·`RecoveringBatchErrorHandler` 대체)
- 기본 동작
    - 기본 BackOff `FixedBackOff(0L, 9)` — 간격 0, 9회 재시도(총 10회 시도)
    - 레코드 리스너: 실패 시 미처리 레코드로 **seek**해서 다시 폴링(재시도), 소진되면 ERROR 로그
    - 배치 리스너: 실패한 레코드 인덱스를 `BatchListenerFailedException`으로 알려줘야 그 지점부터 처리
    ```java
    // 팩토리에 등록: 1초 간격 2회 재시도 후 recoverer로
    factory.setCommonErrorHandler(new DefaultErrorHandler(recoverer, new FixedBackOff(1000L, 2L)));
    ```

## 재시도 간격 — BackOff

- `FixedBackOff(delayMs, maxAttempts)` — 고정 간격 (`FixedBackOff.UNLIMITED_ATTEMPTS`로 무한)
- `ExponentialBackOffWithMaxRetries(n)` — 지수 증가 + 최대 횟수 제한
    ```java
    var bo = new ExponentialBackOffWithMaxRetries(6);
    bo.setInitialInterval(1_000L); bo.setMultiplier(2.0); bo.setMaxInterval(10_000L);
    // 재시도 간격: 1, 2, 4, 8, 10, 10초
    ```

## 예외 분류 — 재시도할 것 vs 바로 복구할 것

- 기본적으로 모든 예외를 재시도하지만, 일부는 **fatal로 분류되어 재시도 없이 즉시 recoverer로** 보냄
- 기본 non-retryable (fatal)
    - `DeserializationException`, `MessageConversionException`, `ConversionException`, `MethodArgumentResolutionException`, `NoSuchMethodException`, `ClassCastException`
    - → [4단계](04.serialization.md)의 `ErrorHandlingDeserializer`가 넘긴 `DeserializationException`이 여기서 바로 복구로 빠짐 (역직렬화 실패는 재시도해도 무의미)
- 조정: `addNotRetryableExceptions(...)` / `addRetryableExceptions(...)` / `setClassifications(map, defaultRetryable)`
- `Error` 서브클래스는 에러 핸들러를 거치지 않음 (RuntimeException 계열만 처리)

## DLT — DeadLetterPublishingRecoverer

- 재시도가 소진된 뒤 실행되는 recoverer — 실패 레코드를 **데드레터 토픽(DLT)** 으로 발행
- 기본 DLT 토픽: `<원본토픽>-dlt`, 원본과 같은 파티션 번호 (그래서 DLT 파티션 수 ≥ 원본 필요)
    ```java
    var recoverer = new DeadLetterPublishingRecoverer(kafkaTemplate);   // 발행에 KafkaTemplate 사용
    factory.setCommonErrorHandler(new DefaultErrorHandler(recoverer, new FixedBackOff(0L, 2L)));
    ```
- DLT 레코드 헤더에 실패 정보 추가: `DLT_EXCEPTION_FQCN`, `DLT_EXCEPTION_MESSAGE`, `DLT_ORIGINAL_TOPIC`/`PARTITION`/`OFFSET`, `DLT_ORIGINAL_CONSUMER_GROUP` 등
- 예외별로 다른 DLT 토픽 지정, 페이로드 타입별 다른 `KafkaTemplate` 사용도 가능

## 논블로킹 재시도 — @RetryableTopic

- 위 `DefaultErrorHandler` 재시도는 **블로킹** — 재시도 동안 해당 파티션이 멈춰 뒤 메시지 처리가 지연됨
- `@RetryableTopic`은 실패 메시지를 **별도 retry 토픽으로 보내고**, 컨슈머는 다음 메시지를 계속 처리 (논블로킹)
- retry/DLT 토픽을 자동 생성·네이밍: `<topic>-retry-0`, `<topic>-retry-1`, …, `<topic>-dlt`
    ```java
    @RetryableTopic(
        attempts = "4",
        backoff = @Backoff(delay = 1000, multiplier = 2.0, maxDelay = 10000),
        include = { RemoteException.class })          // 이 예외만 재시도
    @KafkaListener(topics = "orders")
    public void process(Order order) { ... }

    @DltHandler                                        // DLT로 떨어진 메시지 처리 (같은 클래스 내)
    public void handleDlt(Order order) { ... }
    ```

| 속성 | 용도 |
|---|---|
| `attempts` | 총 시도 횟수 |
| `backoff` | `@Backoff(delay, multiplier, maxDelay)` 지연 전략 |
| `autoCreateTopics` | retry/DLT 토픽 자동 생성 |
| `include` / `exclude` | 재시도할 / 건너뛸 예외 타입 |
| `dltStrategy` | `FAIL_ON_ERROR`(소진 후 DLT) / `NO_RETRY`(즉시 DLT) 등 |
| `topicSuffixingStrategy` | retry 토픽 이름 규칙 |

- 전역 설정은 `RetryTopicConfiguration` 빈으로 (`RetryTopicConfigurationBuilder`)
- 제약: 배치 리스너 불가, 컨테이너 트랜잭션(8단계)과 병행 불가

## 메서드 레벨 에러 핸들러

- 특정 리스너에만 적용하는 `KafkaListenerErrorHandler` — 컨테이너 에러 핸들러보다 먼저, 메서드 호출 실패를 가로챔
    ```java
    @KafkaListener(topics = "myTopic", errorHandler = "myErrorHandler")  // 빈 이름 지정
    public void listen(String message) { ... }
    // (msg, ex) -> { ... 복구값 반환 또는 throw ex }  형태의 빈
    ```
- 컨슈머·`Acknowledgment` 접근이 필요하면 `ConsumerAwareListenerErrorHandler` / `ManualAckListenerErrorHandler` 사용 (단, 에러 핸들러 안에서 seek 금지)
