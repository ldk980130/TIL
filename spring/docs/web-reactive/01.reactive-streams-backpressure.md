# 1. 리액티브 스트림과 배압

- WebFlux를 이해하려면 그 토대인 **리액티브 프로그래밍**과 **리액티브 스트림 명세**부터 봐야 한다
- 동기/블로킹·논블로킹의 기본 개념은 [Blocking IO / Non-blocking IO](../../../os/blocking-io&non-blocking-io.md), [Async·Sync·Blocking·Non-blocking](../../../os/synchronization/async-sync-bocking-nonblocking.md) 참고

## 블로킹은 자원을 낭비한다

- 성능을 끌어올리는 방향은 둘
    - **병렬화**: 스레드·하드웨어를 더 투입 — 자원을 더 쓰는 방식
    - **효율화**: 지금 쓰는 자원을 덜 낭비하는 방식 — 리액티브가 노리는 지점
- 블로킹 호출의 문제: I/O·DB·네트워크 응답을 기다리는 동안 **스레드가 idle 상태로 묶인다**
    - 요청당 스레드(thread-per-request) 모델은 동시 요청이 늘수록 스레드가 선형으로 늘어남 → 메모리·컨텍스트 스위칭 비용 증가
- 병렬화(스레드 추가)는 하드웨어를 채우지만, 동시성 복잡도를 키울 뿐 **대기로 인한 근본적 비효율은 풀지 못한다**
- 같은 문제를 "스레드를 싸게 만들어" 푸는 다른 접근이 [가상 스레드](../../../java/thread/virtual-thread.md)(Loom) — 블로킹 코드를 그대로 두고 스레드 비용을 낮춘다. 리액티브는 반대로 **스레드를 적게 유지하되 블로킹하지 않는** 쪽

## 비동기의 한계 — 콜백과 Future

- 자바의 기존 비동기 모델은 합성(composition)에 약하다

| 방식 | 한계 |
|---|---|
| 콜백(callback) | 결과 도착 시 실행할 함수를 넘김. 여러 단계를 엮으면 깊게 중첩되어 **콜백 헬** — 가독성·에러 전파가 무너짐 |
| `Future<T>` | 즉시 반환되지만 `get()`이 **블로킹**(우발적 블로킹), 지연 평가(lazy) 없음, **다중 값**(0..N) 표현이 어려움, 에러 처리 빈약 |

- 리액티브 라이브러리(Reactor)는 이 한계를 연산자 체인으로 풀어, 비동기 흐름을 **명령형처럼 읽히게** 합성한다

## 리액티브 프로그래밍의 특징

- **데이터를 흐름(flow)으로 다룬다**: 값 하나가 아니라 0..N개가 시간에 따라 흐르는 시퀀스로 보고, **연산자(operator)**로 가공
    - 각 연산자는 이전 단계의 `Publisher`를 감싸 새 `Publisher`를 만든다 → 체인이 곧 처리 파이프라인
- **구독 전엔 아무 일도 일어나지 않는다(nothing happens until you `subscribe()`)**: 연산자를 엮어도 즉시 실행되지 않고, `subscribe()` 시점에 전체 체인이 동작 → 선언과 실행의 분리(지연 평가)
- **합성 가능성·가독성**: 코드 구조가 추상적 처리 흐름의 구조를 그대로 반영
- **배압(backpressure)**: 소비자가 생산 속도를 제어 (아래)
- **hot vs cold**
    - **cold**: 구독마다 처음부터 새로 시작 (예: 구독할 때마다 새 HTTP 요청 발생)
    - **hot**: 구독과 무관하게 진행되며, 늦게 구독한 쪽은 **구독 이후 신호만** 받음
    - (자세한 구분은 Reactor의 `Mono`/`Flux`에서 — 2단계)

## 리액티브 스트림 명세

- JVM에서 리액티브 라이브러리들이 **상호 운용**되도록 인터페이스와 상호작용 규칙을 정한 표준 명세
    - Java 9부터 `java.util.concurrent.Flow`로 표준 라이브러리에 편입 (RxJava·Reactor·R2DBC 등이 이 규약을 구현)
- 단 4개의 인터페이스로 구성
    ```java
    public interface Publisher<T> {
        void subscribe(Subscriber<? super T> s);   // 구독 시작점 (데이터 생산자)
    }
    public interface Subscriber<T> {
        void onSubscribe(Subscription s);           // 구독 확립 — 여기서 request로 수요를 건다
        void onNext(T t);                           // 데이터 도착 (0..N회)
        void onError(Throwable t);                  // 에러로 종료 (터미널)
        void onComplete();                          // 정상 종료 (터미널)
    }
    public interface Subscription {
        void request(long n);                       // 배압: 구독자가 받을 수 있는 개수(demand)를 요청
        void cancel();                              // 구독 취소
    }
    // Processor는 구독자이면서 생산자 — 중간 변환 단계
    public interface Processor<T, R> extends Subscriber<T>, Publisher<R> {}
    ```
- **시그널 프로토콜**: `onSubscribe onNext* (onComplete | onError)?`
    - `onSubscribe` 1회 → `onNext` 0회 이상 → 종료 신호(`onComplete` **또는** `onError`)는 최대 1회
    - 종료 신호 이후로는 어떤 신호도 오지 않음

## 배압 — request(n)

- 핵심은 `Subscription.request(n)`: **구독자가 "지금 n개까지 받을 수 있다"고 수요(demand)를 명시적으로 요청**하고, 생산자는 그 수요를 넘겨 `onNext`를 보내지 않는다
- 효과: 순수 push(생산자가 밀어내기)를 **push-pull 하이브리드**로 바꿈 — 다운스트림이 업스트림에서 n개를 당겨오는(pull) 형태
- 왜 필요한가
    - 동기/블로킹 코드에서는 호출자가 기다리는 것 자체가 **자연스러운 배압** 역할을 한다 (느린 소비자가 빠른 생산자를 막음)
    - 논블로킹에서는 그 제동이 사라지므로, **빠른 생산자가 느린 소비자를 압도(overwhelm)하지 않도록** 속도 제어 장치가 명세 수준에서 필요 → 그게 `request(n)`

## 논블로킹은 곧 리액티브

- **리액티브** = 변화에 반응하는 프로그래밍 모델 — 블로킹되어 멈춰 기다리는 대신, 작업 완료·데이터 도착 **알림(notification)에 반응**한다는 점에서 논블로킹은 그 자체로 리액티브
- 기대 효과
    - **적고 고정된 수의 스레드**와 더 적은 메모리로 확장 → 부하 상황에서 **예측 가능하게** 스케일
    - 서블릿 스택(요청당 스레드)과 달리 동시 연결이 많아도 스레드가 폭증하지 않음 — 서블릿 스택과의 비교는 WebFlux 개요(5단계)에서, 짝이 되는 [Web MVC](../web-servelet/web-mvc.md)도 참고
- 전제와 주의
    - 이점은 **지연(latency)이 있을 때** 드러난다 — 느리고 불규칙한 네트워크 I/O가 섞여야 리액티브 스택의 강점이 부각됨 (CPU 바운드 작업엔 이점이 작음)
    - **체인 전체가 끝까지 논블로킹**이어야 한다 — 중간에 블로킹 호출이 하나라도 끼면 그 스레드가 묶여 이점이 사라짐 (블로킹 코드 격리는 4단계 스케줄러에서)
